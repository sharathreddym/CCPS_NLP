## 1. How the code works (context)

Pipeline: **clean ŌåÆ LLM extract ŌåÆ rules fix**, deployed as an Azure ML online endpoint.

```

   ŌööŌöĆ ner_helper.run_ner()
        Ōö£ŌöĆ pre_processing.data_preprocessing()   (clean the query)
        Ōö£ŌöĆ Azure OpenAI GPT-4.1-mini             (extract 17 entity types)
```

- `init()` loads all reference data into a `DEPENDENCIES` dict (once at startup).

  only on `result["GRADE"]` (never on `COMPETITOR_GRADE`).

## 3. Root-cause analysis

**Confirmed from the askchemille Network tab (`response-nlp`) for `Zytel 101F BK009`:**
```
entities.GRADE: []            entities.COMPETITOR_GRADE: []
outOfScope.active: true
outOfScope.entities.GRADE: [ "zytel 101f bk009" ]
```

The grade landed in `GRADE` and was moved to `outOfScope`. Simulating
`post_processing.identify_out_of_scope_items()` against the real data:

- Normalized `zytel101fbk009` was an **exact member of the out-of-scope `grades` / `gradesExternal`** lists.
- It was **absent from the in-scope lists** (`gradesInScope`, `gradesInScopeExternal`).
- The in-scope master instead had a near-twin **`zytel101f bkb009`** (color `BKB009`, extra `b`).
- The match fired at `post_processing.py:596` (grade is a substring of an out-of-scope entry),
  **before** color-code stripping, so it was flagged for both internal and external users.

**Root cause = data, not code.** The generator notebook de-conflicts via
`out_of_scope_grades = out_of_scope_grades ŌłÆ in_scope_grades`, so a grade is only removed from
the out-of-scope list if it appears in `SPT.PRODUCT_CD`. Because `SPT` only held the `BKB009`
color variant (not `BK009`), the `BK009` entry was never removed ŌåÆ flagged.

**Contrast ŌĆö Celanyl:** `celanylxs3gf60bg1019cef` was already an exact member of the in-scope
lists and absent from the out-of-scope lists ŌåÆ **not flagged** (already correct in the data).

## 4. Two data regressions found & fixed along the way

While restoring the files, the user's first two update attempts were themselves broken ŌĆö worth
recording as gotchas:

1. **32 KB truncation.** Four JSON files were saved through something with a ~32 KB cap
   (clipboard/editor), truncating them mid-string with garbled `""\t ""` tails ŌåÆ
   `json.load` failed ŌåÆ `init()` would crash. Re-copied full files fixed it (real sizes are
   hundreds of KB to MB; the `_index` file is 2.9 MB).
2. **Wrong `abbreviations.xlsx`.** A single-sheet `Sheet1` workbook was added, but `init()`
   reads 4 sheets by name (`PROPERTY`, `FILLER`, `FEATURE`, `common_abb`). Replaced with the
   correct multi-sheet file (367 / 62 / 63 / 39 rows) ŌåÆ full `init()` dry-run then succeeded.

Lesson: after any dependency update, validate that every file `json.load`s / `read_excel`s
and that `outOfScopeData.json` has all 10 keys.
